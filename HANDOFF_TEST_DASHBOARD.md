# Handoff — Editing the Test Dashboard (`test.html`)

Everything you need to start editing Sierra's **staging** dashboard. Read this once; it'll
save you from the two traps that bite everyone.

---

## 1. What it is

**`test.html`** = "Sierra Command" — a single-page **staging** dashboard. It opens on a
**gate landing** (four cards: HVAC·ROPP, CA Sales, HVAC·Service, Plumbing) and switches
between the four dashboard views in-page via `setView(v)`. It's one big file.

- **Live URL:** https://johnschwinghamer94-lab.github.io/sierra-ropp-dashboard/test.html
- **Variants:** `test2.html`, `index_test.html` (older staging copies — ignore unless told otherwise)
- **Repo:** `github.com/johnschwinghamer94-lab/sierra-ropp-dashboard`

> ⚠️ **`index.html` is PRODUCTION — do not edit it.** It's a *precompiled* build and is
> **auto-regenerated** by a cloud pipeline (any hand-edit gets overwritten). Do all work in
> `test.html`. Promoting a change from test → production is a separate step (see §7).

---

## 2. The one thing that makes editing easy

`test.html` is **raw JSX compiled in the browser** by `babel-standalone` (`<script type="text/babel">`).
**There is no build step.** You edit JSX directly, save, and refresh — that's it.

(Production `index.html` is the opposite: precompiled `React.createElement(...)`. So *don't*
copy code straight from `index.html`; write normal JSX in `test.html`.)

**What it loads (all from CDN except engine.js):**
| Script | Purpose |
|---|---|
| React 18 + ReactDOM (UMD, production) | UI |
| `babel-standalone` | compiles the in-page JSX at load time |
| `xlsx` | parse Excel report uploads |
| `Chart.js` | charts |
| `engine.js` (local, in repo) | shared helpers/state (`window.Engine.*`, `window.__*` globals) |

Because Babel compiles on load, first paint is a bit slow — that's expected for staging.

```bash
cd sierra-ropp-dashboard
```

## 4. Preview locally (must be over http, not file://)

The page `fetch()`es `hourly.json` and the GitHub API, so opening the file directly won't work.
Serve the folder:
```bash
cd sierra-ropp-dashboard
python3 -m http.server 8080
# then open  http://localhost:8080/test.html
```
Needs internet (React/Babel/Chart load from CDN). Edit `test.html` → save → refresh the tab.

## 5. How it's structured (orientation)

- **One file.** Landing gate + all four views live in `test.html`.
- **Navigation:** `setView(v)` swaps the active view; the gate "ENTER" buttons call it.
- **Data:**
  - Gate-card KPIs (e.g. `$16.00M`, `44.6%`) are **seed literals** in the file — they can go
    **stale**; they're not live. Fine for layout work; if you're changing numbers, know they're hardcoded.
  - Live-ish data comes from `fetch("hourly.json")` (hour-by-hour) and the GitHub API (`r.json`).
- **Styling:** inline `<style>` + inline React styles. Dark theme.

To find a view's code, search `test.html` for its label (e.g. `CA SALES`, `setView`).

## 6. Deploy (publishing your change)

It's a static file — **no build**. Commit and push to `main`; GitHub Actions (`deploy.yml`)
publishes to Pages within ~1–2 min.

```bash
git pull --rebase          # ALWAYS pull first — see the trap below
# ...edit test.html...
git add test.html
git commit -m "Test dash: <what you changed>"
git pull --rebase          # again, right before pushing
git push
```
Your change is live at `.../test.html`.

---

## 7. Traps (both have burned people here)

1. **The repo pushes itself constantly.** A cloud pipeline commits data files (`hourly.json`,
   `livefeed.json`, etc.) every few minutes. If you push without pulling first, you'll get
   rejected or conflict. **Always `git pull --rebase` immediately before you push.** Conflicts
   should only ever be on data JSON files (take theirs) — never on `test.html` unless two people
   edit it at once.

2. **Never hand-edit the machine-written files:** `index.html`, `data.json`, `hourly.json`,
   `livefeed.json`, `scorecards.json`, `hourly_state.json`, `livefeed_state.json`. They're
   regenerated by the pipeline and your edits will vanish (and can break the live site).

**Promoting test → production:** don't just copy `test.html` over `index.html`. Production is
precompiled and pipeline-managed. Coordinate with John — promotion is a deliberate, separate step.

---

## 8. 60-second quick start

1. Get collaborator access; `git clone`.
2. `python3 -m http.server 8080` → open `http://localhost:8080/test.html`.
3. Edit the JSX in `test.html`, save, refresh — iterate.
4. `git pull --rebase` → `git add test.html` → commit → `git pull --rebase` → `git push`.
5. Check it live at `https://johnschwinghamer94-lab.github.io/sierra-ropp-dashboard/test.html`.

Questions on a specific view's data wiring → ask John; the gate KPIs and some feeds are
hardcoded/seeded and he knows which are meant to be live.
